Remarks
A rotAppObject is a base class to streamline data access so any object that gets populated with data from the database either wholly or partially should inherit this class. This class has five main purposes:
  1. Provide a consistent interface.
  2. Eliminate tight coupling between the business objects and the data manager.
  3. Abolish redundant code.
  4. Encapsulate change tracking.
  5. Allow the use of polymorphic functions

1. Consistent Interface

Every object that inherits from rotAppObject will behave exactly the same for all the predefined functions. Functions to work on any of the objects based on rotAppObject can seamlessly handle any and all of them including new ones without needing a recompile as long the input parameter's type is the base class and not the derived.

2. Data Manager oblivious to specific objects

The data manager (data access layer) can populate every application object that uses this as its base without any reference to any other object than rotAppObject.

3. Code reuse

Housekeeping and other basic tasks are all done here and done once. Create a new object and it automatically does some of the basics. Also existing functions that take a rotAppObject as a parameter can be used with a new object. See #5 below.

4. Encapsulate Change Tracking

If the flag is set for change tracking then this class will handle all the client side duties, the server still needs to be properly set up. Having all the change tracking code in one place greatly assists in maintenance, removes repeated code from the derived objects making the code specific to each object or domain easier to read and modify.

5. Polymorphic functions

One function can handle an unlimited number of object types, even objects that did not exist when the namespace containing the function was compiled as long as they all derive from the same base class.

All summed up it equals lower maintenance cost and quicker development time.
