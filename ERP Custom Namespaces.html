ERP Custom Namespaces

Our namespaces are analogous with our assemblies in that there is one logical task per. The tblSuppManTranslate are simple enough not to warrant an additional namespace. So in our code and documentation, an assembly is equal to a DLL which is equal to a namespace.
Customized usually refers to an object that was supplied by intuitive and changed by us. Custom usually means an object we designed in house.
Naming conventions used: modified form of hungarian notation with suffixes.

Custom or Customized

Custom

A custom object is one completely designed in house. May or may not interact with vendor supplied objects.

Customized

A vendor supplied object that has been modified.

Custom prefixes

Intuitive

Supplied object. It is also present on an object that is derived from an object that inherits from a supplied base object (i.e. customized object). Used in directories , projectly scoped files , assemblies , and namespaces .


Rotair

Custom built object. Used (same as Intuitive) in directories, projectly scoped files, assemblies, and namespaces.

ims

Supplied class (can be a base class or a derived class) or a custom class derived from a supplied class (base or derived).

rot

Custom class. Can be a base class or a custom class derived from a custom base class.

AO

Indicates an AppObject.

UI

Indicates a form.

Dialog

Windows form that is to be used as a dialog only. Used only in our custom dialogs, Intuitive supplied dialogs are prefixed with UI just like the forms.

HC

Class for printing Hard Copies.

Suffixes

_CUSTOM

An extended supplied object. The method for customizing dictates that we inherit and name the new object the same name except add a suffix of _CUSTOM. Although this is only required on the object level we have carried over the same convention to directories and files. So if we extend the Item object which lives in IntuitiveAO_Item.sln it becomes IntuitiveAO_Item_CUSTOM.sln.

_Collection

A collection of AppObjects.

Remarks

There are dependency charts on some pages. Those charts were made with software that analyzed the compiled assemblies and because of this the software charts the runtime dependencies. It can miss some dependencies that are only required during compile time. For example, any assemblies that are referenced solely for constants residing in that assembly are not needed during run time since during compilation the compiler will optimize the code and copy the constants directly into the compiled assembly thereby eliminating the dependency during run time. When I find any of these I include them in the table.
